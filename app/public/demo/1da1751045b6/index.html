1我們對這個需求的理解

您目前的分析架構是一套典型、而且整理得很乾淨的微軟資料倉儲：

  SQL Server（ERP）              營運系統，所有原始資料的源頭
        │
        ▼
  SQL Server（資料倉儲）          整理成分析用的倉庫，1 個事實表 + 8 個維度表
        │
        ▼
  SQL Analysis Services（SSAS）   把資料預先彙總，給報表快速查詢

您希望把這套架構搬上 Google BigQuery，改成雲端資料倉儲。我們把它拆成三層來看，因為這三層的難度與風險差很多，講清楚對您決定範圍最有幫助：

  1. 資料層（搬資料本身）：

    把 1 個事實表 + 8 個維度表的結構與資料搬到 BigQuery，並建立一條每天自動把新資料送上雲的同步管線。您的規模乾淨明確，這一層是可控、可驗證的。

  2. 分析層（SSAS 那塊）：

    BigQuery 沒有 SSAS 的對應物，原本 SSAS 裡的彙總與計算邏輯若要保留，需要在 BigQuery 上以另一種方式重建。這一層的工作量，取決於 SSAS 裡累積了多少計算邏輯。

  3. 下游層（誰在用這個倉儲）：

    原本連著這個倉儲的報表或程式，遷移後需要重新指向 BigQuery。這一層取決於您這邊的下游是哪一種。

我們的判斷：真正的價值與風險，不在「搬資料」，而在第 2、3 層。搬資料反而是最確定的部分。所以這份提案會把範圍清楚地切在「資料層」，把第 2、3 層的處理方式講明白，讓您先用最低風險完成核心遷移，再決定要不要把上層一起做。

2為什麼是我們

資料庫遷移與雲端基礎設施建置，是我們實際做過、交付上線的工作。

我們處理過正式環境的資料庫搬遷與雲端架構建置，也維運過生產級的資料同步管線，知道一場遷移真正會出問題的地方在哪。

這個案子的本質，正是我們熟悉的遷移問題：

一場資料倉儲遷移的關鍵我們的對應經驗
把既有資料庫完整、無誤地搬到新環境做過正式環境的資料庫遷移
在雲端把基礎設施與權限架好做過雲端基礎設施建置
建一條每天自動更新的資料同步管線維運過生產級的資料同步
遷移後驗證新舊資料一致遷移收尾的對帳，是我們本來就會做的事

遷移的方法論是跨雲通用的：全量搬遷、增量同步、切換前的逐表對帳。這套流程不會因為目的地從一朵雲換成另一朵雲而改變，差別只在換用對應的工具。

BigQuery 作為一個全託管的雲端倉儲，遷移要點與我們做過的資料庫遷移一致。您的規模（1 事實表 + 8 維度表）乾淨明確，我們有把握把資料層做穩、做對。

3本案的交付範圍（資料層）

我們建議把這次的範圍切在「資料層」，這是可以保證做對、而且可以用數字驗證的部分：

#交付物內容
1BigQuery 倉儲建置在您的雲端專案上，依現有的事實表與維度表結構，建立對應的 BigQuery 資料表，並處理好型別與精度的對應
2歷史資料全量遷移把現有的全部歷史資料，完整搬進 BigQuery
3每日自動同步管線建立一條管線，每天自動把 ERP 端產生的新資料更新到 BigQuery，不需要人工搬運
4資料一致性對帳遷移完成後，逐表比對新舊兩邊的筆數與關鍵金額／數量總計，提供一份對帳結果，讓「資料搬對了」是可驗證的，而不是口頭保證
5切換與交付文件把下游從舊倉儲改指向 BigQuery 的切換步驟、同步管線的運作說明、後續維護要點，讓您這邊日後可自行維護。多數情況下切換只是在報表工具裡改資料來源，不需要改動您現有的程式（僅當有應用程式以內嵌語法直連倉儲時，該程式才需適配，屬第五節範圍外）

4有一件事我們想先誠實跟您說：數字一致性

這是真正懂這類遷移的人才會主動提醒您的一點，我們想講在前面。

SQL Server 與 BigQuery 在一些底層行為上有細微差異，例如數值的小數精度、整數相除的處理、空值在加總與排序時的行為、日期時間的時區處理。這些差異多半不會報錯，卻可能讓搬過去的數字默默差一點點，最後反映成「報表金額跟舊系統差了幾塊錢」。

我們處理這件事的方式很直接：把「逐表對帳」當成交付的一部分，而不是做完才檢查。我們會逐表比對新舊兩邊的筆數，以及關鍵的金額、數量總計，確認兩邊一致才算這一層完成。這份對帳結果會交給您，是您驗收「資料層做對了」的客觀依據。

一句話：我們保證的是「資料一致，且可對帳驗證」，這是我們能負責、也能證明的部分。

5我們建議先劃在範圍外的部分（盤點後另行報價）

承上，有三塊工作的份量，取決於您這邊現況的細節，在還沒看到實際內容前無法準確估算。我們建議先不放進這次的固定範圍，等盤點清楚後，您再決定要不要加做、我們再給對應報價。這樣對您最公平，也避免把不確定的工作硬塞進固定價裡：

  1. SSAS 分析層的重建：

    SSAS 是綁在微軟生態的分析引擎，無法直接搬到 BigQuery（兩者是不同的技術）。但它算出來的彙總與 KPI 結果，我們可以在 BigQuery 上以等價方式重建，並逐項對帳驗證數字一致，也就是「保留的是結果，重建的是實現方式」。這部分的工作量，取決於 SSAS 裡有多少計算邏輯，而這些邏輯通常沒有文件、只存在於 SSAS 的設定裡。我們需要先取得 SSAS 的設定內容盤點過，才能準確報價。

  2. 既有報表的重新接線或重建：

    原本連著這個倉儲的報表（例如 Power BI、Excel 樞紐分析），遷移後需要重新指向 BigQuery。如果報表原本是連一般資料表，多半是調整連線設定；如果報表原本是連 SSAS，則需要在 BigQuery 上重建對應的資料模型。實際工作量取決於報表數量，以及它們原本連的是資料表還是 SSAS。

  3. 下游程式的連線與語法適配：

    如果您這邊有應用程式直接連這個倉儲撈資料，遷移後該程式的連線方式與內嵌的查詢語法需要做對應調整（SQL Server 與 BigQuery 的連線方式與部分語法不同）。這部分取決於有沒有這類程式、有幾支。

這三塊不是我們不做，而是它們的份量現在看不到。先盤點、再報價，您不會替看不見的工作買單。

6關於 POC 與雲端成本

我們會以一個可實際驗證的 POC（先把資料搬上去、跑通同步、完成對帳）作為第一階段的交付。關於成本，先講清楚讓您安心：

  • POC 與正式環境，都直接建在您方的 Google 雲端專案上。雲端資源與帳單歸您方，我們不經手。這樣做的好處是：POC 驗證過的環境，就是日後的正式環境，不需要再搬一次、不會重工。
  • 這個階段的雲端費用很低。資料的搬遷與每日同步、以及驗證用的查詢，用量都很小，通常落在 BigQuery 的免費額度內。唯一會實際產生費用的是資料的儲存，而它取決於您事實表的大小；以這類規模，即使超出免費額度，每月儲存費用也僅在極小的範圍。實際金額我們可在看到資料量後為您估算。
  • 您這邊需要先準備的，是開好一個 Google 雲端專案、設好計費帳戶，並讓我們能存取。這部分我們可以協助您一步步設定。

7報價前想先跟您確認的事項

以下幾點會直接影響範圍與工作量，想先跟您確認，以利於對齊認知：

  1. 遷移時是否需要做到「不中斷」？

    這個倉儲是給分析報表用的，一般而言可以挑離峰時段做切換，使用者幾乎無感，不需要中斷您的 ERP 營運。想先跟您確認：您這邊對切換時的服務中斷，是可以接受短暫離峰切換，還是有「完全不可中斷」的硬性要求？這會影響做法與報價。

  2. SSAS 那塊，您打算保留多少？

    先說明一個常被誤會的點：SSAS 本身無法直接搬到 BigQuery（兩者是不同的技術），但它算出來的報表與 KPI 結果，我們可以在 BigQuery 上以等價方式重建、並對帳驗證數字一致。想先了解：這些既有的彙總與 KPI，您希望完整保留（我們逐項在雲端重建到結果一致），還是趁這次遷移精簡成真正常用的那幾項？這會決定第五節第 1 點的份量。

  3. 現在是誰在用這個倉儲、用什麼工具？

    下游的報表或程式是哪幾種（Power BI、Excel、其他 BI 工具，或是某支內部程式）？它們原本是連一般資料表，還是連 SSAS？這決定第五節第 2、3 點的範圍。

  4. 來源端的存取與資料樣態：

    我們需要能連到來源的 SQL Server 來做遷移與同步。您這邊的資料表裡，是否有可靠的「更新時間」或主鍵欄位可用來判斷增量？以本案規模，即使沒有，我們也可以用每日全量重載穩定處理，這題只是讓我們挑最合適的同步方式。

9時程

自取得來源資料庫存取與 Google 雲端專案存取後：

  • 核心遷移（建置 + 全量遷移 + 對帳 + 切換）：約 7～10 天，您能實際在 BigQuery 上看到搬好的資料、對帳結果，並完成切換。
  • 自動同步管線：約 1 週至 1 個月，視您期望的同步方式而定。每日批次更新較單純、可快速交付；若期望更即時或更複雜的同步，所需時間會往上。我們會在第七節第 4 點確認後給您明確時程。
