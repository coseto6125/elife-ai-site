1我們對這個需求的理解

貴公司目前的作業流程是：收到各家供應商的報價單（格式不一），由人工讀出料號、品名、規格、數量、單價、交期、幣別等欄位，自行計算成本（加成、運費、匯率、稅），再依固定格式整理出成本分析表與對客戶的報價單。這條流程裡，真正花時間且容易出錯的有三段：

  • 讀供應商報價單：每家格式都不一樣，欄位位置、單位寫法、幣別各異，人工逐張判讀既慢又易誤植。
  • 算成本：加成比例、匯率、運費、稅務攤提，公式雖固定，但每張手算仍有出錯空間，且改一個參數要全部重算。
  • 出客戶報價單：成本確定後，再依固定格式重新排版成對外報價，等於同一筆資料謄打兩次。

因此本案的核心是：打造一套工具，把「供應商報價單進來 → 辨識資料 → 自動算成本 → 自動產出客戶報價單」這整條人工作業自動化，並在資料異常時主動標記。您在需求裡已點出關鍵分法：約 70% 的欄位因常配合固定供應商、格式穩定，可寫固定規則直接判讀；其餘 30% 需串接外部 API 取得。我們完全認同這個分層，並以此為基礎設計方案：穩定的多數用最精準零成本的固定規則處理，少數變動的才向外部 API 取值，讓系統長期可用、且不增加校正人力。本案的辨識、計算、輸出三段本質上多是確定性的規則邏輯，我們不為了導入 AI 而導入；只有在報價單格式真的發散到規則難以窮舉、或遇上掃描影像時，才作為可選的輔助手段。

3兩段固定輸出：成本分析與客戶報價單

本案與單純「抽取」案的差別在於：辨識資料只是第一步，真正的價值在後面兩段固定格式的自動產出。我們的做法如下：

您的需求我們的具體做法
產生成本分析（固定格式）辨識出供應商單價、數量、幣別後，系統依您既有的成本公式自動計算：成本加成、運費攤提、匯率換算、稅務等，輸出固定格式的成本分析表。公式參數（加成率、匯率來源）集中可設定，調一處即全部重算，不必逐張手改。
產生客戶報價單（固定格式）成本確定後，依您對外的固定報價格式自動排版產出，同一筆資料不再謄打第二次。可一鍵由成本分析帶出對客戶報價，並保留兩者對應關係以利日後核對。
資料異常主動標記每次執行後自動產出處理報告：本批成功幾張、異常幾張、失敗哪幾張及原因（格式不符、欄位缺漏、API 取值失敗等）。異常不中斷整批作業，問題集中列示，一覽即明。

自動規則驗證（確定性檢查，不需 AI）

系統對每筆讀取值與計算結果自動執行四項規則檢查，這些都是明確可定義的規則，不依賴 AI 判斷：

  1. 公式核對

    單價×數量是否等於金額、加成與匯率換算是否正確、成本與報價是否符合既定公式。

  2. 交叉一致性

    同一張報價單內欄位相互對照，成本分析與客戶報價單金額是否能對得起來。

  3. 必填／完整性

    應有欄位是否缺漏、讀取結果是否為空值或亂碼、API 是否成功取回。

  4. 極值／異常標記

    數值落在合理範圍外（負數、極端值、匯率或日期超出範圍）者即予標記，送末端複核。

簡言之，本工具是讀取 → 自動算成本 → 自動出報價單 → 僅異常項目需人工複核的完整閉環，驗證靠的是明確規則而非 AI。對貴公司而言，是節省每張報價單的重複判讀與謄打，而非增添校正人力。

4資料從哪裡來：A／B／C 三種來源（呼應您的 70%／30%）

這一段釐清「每個欄位的資料從哪裡取得」，直接對應您需求中「70% 固定欄位、30% 串接 API」的分法。A、B、C 是三種資料來源，依各欄位實際情況採用、亦可在同一張報價單上並用：

來源這些資料在哪怎麼取得
A. 報價單本身（約 70%）資料就在供應商報價單檔案裡，常配合的供應商格式穩定直接讀取檔案欄位（Excel／CSV 等結構化檔），或以固定規則（如 regex）對位取得料號、品名、數量、單價、幣別等
B. 報價單裡有、但格式偏差同樣在報價單內，只是欄位錯位、合併儲存格、單位寫法不一，或為掃描影像以針對性程式碼校正回標準格式
C. 報價單裡沒有、需向外部取得（約 30%）報價單上沒有、得另外查的資料：料號規格、即時報價、匯率等向外部串接 API 取得
C 的工作量與成本，取決於那 30% 的「API」具體指什麼。這是本案最需先釐清的一點：是向供應商或第三方平台查即時報價／料號規格？是匯率 API？還是要接貴公司自家的 ERP／系統？不同來源在串接難度、是否需付費或授權金鑰、回應穩定度上差異很大。我們會在第六節把這點列為正式報價前提，看過實際 API 文件後才能將工作量講準。
搭配方式：同一套工具中三種來源並存混用，報價單本身有的採 A、格式偏差的用 B 校正、報價單沒有的才以 C 向外部取得，彼此互不衝突。實際各佔比重，依貴公司供應商報價單的實際樣貌而定。

AI／OCR 是可選方案，不是非用不可

A、B 講的是「資料從哪來」；至於「怎麼把它讀出來」，多數情況下直接讀檔案欄位或固定規則就夠了，不需要 AI。AI 與 OCR 是橫跨 A／B 的可選辨識手段，只在以下情形才動用，且皆可配合一般／輕量 CPU 的電腦運行、無需獨立顯卡，資料留存本機：

  • 掃描影像才需 OCR：若報價單是掃描或拍照影像，沒有可直接讀取的欄位，才以成熟的輕量繁中辨識引擎讀出文字。純 Excel／文字檔則用不到。
  • 格式真正發散時才用小模型：若報價單格式發散到固定規則難以窮舉，才以可在一般 CPU 運行的輕量模型補足長尾欄位。供應商若多為固定模板，這一項通常用不到。
我們的取向是能用固定規則就不用 AI：把確定性的工作交給確定性的方法，準確且零成本、結果可重現；只有掃描影像與真正發散的長尾才動用 AI／OCR 輔助。實際是否需要、以及引擎選型，將於評估報價單樣本後確定。

6報價前需確認的事項

以下事項直接影響工作量與辨識準確率，懇請先行確認，此亦為本類案件成敗之關鍵：

  1. 能否提供部分去識別化的真實供應商報價單範例？

    此為最關鍵之素材。經評估實際檔案，方能判斷格式固定程度、是否需 OCR、固定規則可涵蓋之範圍。範例越多、越能涵蓋各供應商寫法，上線正確率越高。

  2. 那 30% 的「串接 API」具體指什麼？

    這是本案報價最關鍵的變數。是向供應商或第三方查即時報價／料號規格？是匯率 API？或接貴公司自家 ERP／系統？是否已有 API 文件、是否需付費或授權金鑰？懇請提供 API 文件或說明，我們才能將這部分工作量講準。

  3. 供應商報價單為 Excel／Word／純文字 PDF，抑或掃描影像？

    結構化的 Excel 命中率最高；純文字 PDF／Word 可直接讀取；若為掃描或拍照件，則需 OCR，工作量有所不同。

  4. 成本分析的計算公式為何？

    加成比例、運費與稅務攤提方式、匯率取得來源（固定值、每日更新、或由 API 取得）。懇請提供現行的成本分析表範例（欄位與公式）。

  5. 客戶報價單的固定格式為何？

    懇請提供現行對客戶的報價單範本（Excel／Word／PDF），含欄位順序、版面、是否需公司抬頭與印章欄位等。

  6. 運行於何種設備、是否需排程？

    單機桌上型電腦即可，抑或需排程自動處理？設備之 CPU／記憶體規格（若需用到掃描件 OCR，會影響其可運行規模）。
