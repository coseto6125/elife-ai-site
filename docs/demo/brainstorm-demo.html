需求釐清對話流 — 理想長相

Brainstorming 式漸進釐清：一次一問、優先選擇題、依序問 purpose → constraints → success， 進入需求階段時把模型升級為 Opus 4.8。

e-life-ai 小幫手 model: claude-haiku-4-5
小幫手
嗨～想做點什麼都可以跟我說，哪怕只是一個方向 🙂
訪客
我想做一個訂房系統
偵測到需求描述 → 進入 requirement_brainstorm · 模型升級 claude-haiku-4-5 → claude-opus-4-8
小幫手 · opus 4.8
purpose 好喔！訂房系統有很多種樣子。先抓個方向就好——這比較像哪一種？
民宿 / 旅館自己的訂房頁 跨多家的訂房平台 內部管理用的後台
訪客
我自己一間民宿，想讓客人線上訂房
小幫手 · opus 4.8
constraint 了解～單一間民宿的線上訂房。那金流這塊呢，想收線上付款，還是先讓客人送出申請、你再人工確認就好？
要線上付款（刷卡 / 第三方） 先送申請、人工確認
訪客
先人工確認就好，之後再說
小幫手 · opus 4.8
success 清楚了。那對你來說，這個系統上線後最想解決的是什麼——是減少 LINE 上來回問空房的時間，還是想要一個能對外展示的正式訂房頁？這會影響我們建議的做法。
訪客
主要是不想一直在 LINE 回空房，太花時間
小幫手 · opus 4.8
抓到重點了——核心是「自動擋掉已訂走的日期、客人自己看空房送單」。 這規模其實一個輕量訂房頁 + 日曆管理就能解，不用做大平台。 要不要我幫你安排一次免費的需求釐清對談，把範圍和大概時程談清楚？或先留個聯絡方式都行 🙂
purpose 為什麼做 / 想解決什麼 constraint 限制 / 條件 success 怎樣算成功

要點：① 一次只問一個問題 ② 優先給選擇題降低訪客負擔 ③ 依序釐清 purpose → constraint → success ④ 偵測到需求描述才切 Opus（FAQ / 閒聊維持 Haiku，控成本）⑤ 理解夠了才給方向、軟性引導留聯絡， 全程不在此硬報價。
